## 1. 产品概述

### 1.1 产品定位
面向**测试工程师与开发工程师**的**内网私有化 Web 平台**，通过 AI 辅助生成、管理与执行 API / Web UI 自动化测试用例，解决测试效率低、批量管理难、团队协同混乱的问题。

### 1.2 核心价值
- **提效**：基于 PRD、用户故事、Swagger 文档等，AI 自动生成可执行 Python 测试脚本。
- **统一管理**：项目制隔离的共享用例库，支持评审流程。
- **自动化执行**：服务端集中执行，集成 pytest + Playwright / HTTP 客户端，产出 Allure 详细报告。
- **团队协同**：多人评审、通知分发、Postman 兼容，降低协作成本。

## 2. 用户与场景

### 2.1 目标用户
| 角色       | 职责                               | 核心诉求                                 |
| ---------- | ---------------------------------- | ---------------------------------------- |
| 测试工程师 | 主要操作用户，编写、评审、执行用例 | 减少重复脚本编写，快速覆盖接口和 UI 场景 |
| 开发工程师 | 参与用例生成，关注 API 测试        | 利用 Swagger 直接生成用例，快速验证接口  |
| 项目管理员 | 管理项目成员、评审流程             | 控制用例质量，观测整体执行趋势           |

### 2.2 典型使用场景
1. **接口测试**：测试工程师导入 Swagger/OpenAPI 文档，AI 自动生成单接口参数组合及业务流用例，编辑后提交评审，通过后加入测试计划定时执行。
2. **Web UI 测试**：产品提交 PRD 或用户故事，AI 生成基于 Playwright 的测试脚本，工程师补充页面元素定位，评审后执行并自动截图、生成报告。
3. **提测回归**：开发提交代码后通过 Webhook 触发平台执行预设计划，结果通过钉钉群通知。
4. **Postman 迁移**：团队将积累的 Postman Collection 导入平台，转为规范用例并纳入 AI 增强管理。

## 5. 非功能需求

### 5.1 性能
- 单个项目支持 5000+ 用例，页面列表加载 < 2s。
- 并发执行任务数：同一项目同时最多 3 个执行任务，排队等待。

### 5.2 安全
- 所有传输使用内网 HTTPS（可选）。
- 用例脚本仅在后端执行沙箱内运行，不允许访问公网（除被测服务）。
- API 接口必须鉴权，跨项目数据完全隔离。

### 5.3 可用性
- UI 交互遵循 Element Plus 规范，支持 1920×1080 及以上分辨率。
- AI 生成失败时有明确错误提示，提供重试。

### 5.4 可扩展性
- 后端模块化设计（Django apps），API 测试与 Web UI 测试解耦。
- AI 模型可通过配置切换，不影响业务逻辑。
- 执行节点可注册扩展，支持未来多机分布式执行。

### 5.5 部署与运维
- 提供 Docker Compose 一键部署脚本。
- 依赖组件：Django、Celery、Redis、PostgreSQL、MinIO。
- AI 服务需内网出网权限（仅 HTTPS 访问 LLM API 域名）。

## 6. 数据模型概览

### 6.1 核心实体关系
```
User ──< ProjectMember >── Project
Project ──< ApiEnvironment
Project ──< ApiTestCase
Project ──< UIEnvironment
Project ──< UITestCase
TestCase (基类) ──< ExecutionResult
Project ──< ExecutionPlan (测试计划)
ExecutionPlan ──< ScheduledTask (定时任务)
```

关键字段示例：
- **ApiTestCase**：method, url, headers, body, assertions(JSON), status, reviewer, script(生成后的Python代码)
- **UITestCase**：target_url, browser, script(Python代码), status, reviewer, page_elements
- **ExecutionResult**：case_id, plan_id, status(PASS/FAIL/ERROR), duration, logs, screenshot_urls, request/response_data, allure_uid

## 9. 附录：关键交互示意（文字版）

### 9.1 AI 生成流程
1. 用户在用例库点击“AI 生成”。
2. 选择输入类型：Swagger / PRD文本 / 用户故事，上传文件或填入内容。
3. 平台调用 LangGraph 工作流，前端实时显示“正在生成测试场景...”。
4. 生成完毕后，代码展示在 Monaco Editor 中，用户可编辑。
5. 点击“保存”，用例状态为“待评审”。
6. 评审人收到通知，进入评审页，可查看脚本与需求对比，选择“通过”或“驳回”。
7. 通过后状态变为“可执行”。

### 9.2 执行与报告流程
1. 在“测试计划”中选择多个用例，绑定环境，保存计划。
2. 手动点击“执行”，或设置 Cron 表达式定时执行。
3. 后端马上创建 Celery 任务，用例开始执行；前端可实时看到日志流。
4. 执行结束，生成 Allure 报告，通知发送到群机器人。
5. 用户打开报告，可查看每个用例的步骤、截图、请求响应。
